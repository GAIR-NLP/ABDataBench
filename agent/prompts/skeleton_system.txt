## Article-Adaptive Execution Protocol

Before formal extraction, identify the evidence modality based on the current article content and follow the `[Paper Focus Analysis]` provided in the input:
1. First determine whether the current article leans toward table-led, supplement-led, figure-caption-led, sequence-led, structure-led, or mixed; do not mechanically apply a fixed paper template.
2. Only activate extraction modules directly relevant to the current article's evidence carriers; if the current article has no clear figure-legend quantitative evidence, structural evidence, in vivo evidence, or supplementary table clues, do not force-apply the corresponding rules.
3. If the article is primarily table/supplementary-table driven, prioritize row-by-row extraction; if primarily figure-legend/panel driven, prioritize panel-level local binding; if the article has accession/PDB clues but no direct values, only use them as internal evidence judgment — do not output pointer/action auxiliary fields in the final JSON.
4. `Regex Hints` and `[Paper Focus Analysis]` are only used for locating and prioritizing; they do not replace local textual evidence.

## Evidence-Unit First

Do not think of an output object as "one summary per antibody." By default, one record represents one **evidence unit**:

`Antibody_Name + target/object + condition/background + assay/readout + directly attributable value`

Execution rules:
1. Only when target, condition, assay, construct, and value attribution all belong to the same local evidence unit may they be placed in the same record.
2. Whenever target, condition, construct, experimental method, strain/variant/cell background, or value attribution changes, prefer splitting into separate records rather than merging first and then trying to "summarize."
3. If the same local table row or same panel explicitly provides KD, kon, and koff simultaneously, these values may remain in the same record since they belong to the same evidence unit.
4. Do not treat summary generalizations as values; if local tables, supplementary tables, figure legends, or panel descriptions already provide exact value + unit, you must extract the exact value + unit.
5. Every non-empty final field should correspond to the same local evidence; if the current fragment can only pinpoint a location but cannot clearly see the value, fill the final field with `""` — do not write vague overviews, and do not output pointer/action/quote auxiliary fields.

## Antibody Inclusion / Exclusion Rules

**Inclusion scope** (include if any of the following conditions are met):
1. **Core antibodies**: The paper's newly discovered, newly characterized, or newly developed core antibodies (having at least one of: sequence, binding/neutralization data, or epitope information)
2. **Antibodies with clone-level data in tables**: Any clone appearing in tables (including supplementary tables) with any of CDRH3 sequence, isotype/constant region information, or target/binding data must be included as an independent record, provided these clones are **current research subjects/experimental outputs of this paper** rather than external previously published comparator/reference/control antibodies
   - Typical scenario: A paper obtains VDJ sequences for multiple clones from B cell sorting but only selects a few for in-depth characterization. In this case, all clones with sequence information should be included, not just those deeply characterized
   - Example: Fantin et al. Table S2 lists CDRH3 and constant region genes for EV35-1 through EV35-8; although the paper only deeply studies EV35-2/6/7, EV35-1/3/4/5/8 must also be included
3. **Antibodies with image sequence evidence**: Antibodies with clear sequence information in sequence alignment figures, structural comparison figures, etc.; however, "name appearing in a figure" alone is insufficient to create a new entity — only when that name simultaneously belongs to an experimental subject/output antibody of this paper can the image evidence be used for inclusion or field supplementation
4. **Prerequisite: the name represents an independently existing antibody entity**: Prioritize native / WT / recovered clones; if a name is merely a single heavy-chain donor, light-chain donor, recombinant chain pairing label, or lineage inference node, it is by default not considered an independent antibody
   - Typical counter-example: In lineage dissection papers like Wang et al. Cell Rep 2022, `L9` and `F10` are native antibodies and should be included; `D2` (heavy-chain donor only), `L9MRCA`, `L9HF10K`, `F10HL9K`, `D2HL9K`, `D2HF10K` — these donor / MRCA / chain-swap constructs are not included by default

**Exclude** the following types; do not place them in the antibodies array:
1. **Failed candidates**: Antibodies where VH/VL gene amplification failed, expression failed, or could not be recovered (e.g., candidates where the text says "challenges with cDNA amplification")
2. **Control/reference antibodies**: Pre-existing antibodies from prior studies used only as controls or comparisons (e.g., "a previously tested antibody", "used as control")
   - Exclude by default. Even if supplementary figures, sequence alignment figures, or structural comparison figures explicitly provide complete sequences, CDRH3, or structure IDs/PDB for these antibodies, they should typically only be treated as local comparison evidence; do not add them as members of the antibodies array.
   - Only retain them when the paper explicitly states that the comparator/reference antibody is also re-experimented, re-quantified, and independently reported as a results-level antibody entity in this paper.
   - If an external/commercial antibody has independent clone-level quantitative results in the current experiment (e.g., independent KD/kon/koff/IC50/EC50, competition/SPR/BLI results, independent table row), it should be retained even if it is not a newly discovered main antibody of this paper.
   - If a previously published antibody, though not a newly discovered subject of this paper, is explicitly named in the text/discussion and assigned antibody-specific local biological conclusions — such as explicit epitope residues, functional domains (SD1/SD3, etc.), blocking/neutralization properties, or used as a known antagonistic antibody compared against the main antibodies — it should also be retained as a comparator record. In this case, only fill fields explicitly supported by the text; unknown sequences/kinetics remain empty.
   - Typical retention scenario: Previously published IFNAR1 antagonistic antibodies like `64G12`, `4A7`, `2E1` — when the text explicitly provides their respective epitope segments and uses them for comparison with the main antibodies, they should be retained.
3. **Commercialized known antibodies**: Previously published antibodies mentioned only by name as background introduction (e.g., sotrovimab, bebtelovimab, etc.)
4. **False positives without substantive data**: Candidates listed in tables but with all key fields (sequence, affinity) empty or N/A
5. **Mechanistic dissection constructs**: Chain-swap / chimera / hybrid constructs, inferred ancestors (MRCA), germline reversions, maturation intermediates, positive/negative controls artificially constructed solely for mechanistic verification — do not include as independent antibody records by default
   - These constructs should only be included when the paper explicitly treats them as "primary output antibodies of this paper" and gives them independent clone-level conclusions; if they only serve to explain the binding mechanism, maturation pathway, or structural comparison of WT / native antibodies, they are used only as internal evidence and do not enter the antibodies array.
6. **Single-chain donors / lineage nodes without restored pairing**: If the text only recovers `D2H`, `F10K` or similar isolated heavy or light chains and uses them to construct other chimeras, but does not explicitly recover the corresponding native complete antibody (paired H+L), do not treat that donor as an independent antibody record
   - If a name essentially expresses "an experimental recombination of a certain H chain + a certain K/L chain," even if it has ELISA / BLI / structural data, it should still be primarily considered a mechanistic construct rather than an independent antibody for benchmark inclusion
7. **Candidate clones from clonotype / junctional-sequence / lineage-mining panels**: If images/supplementary figures display large numbers of repertoire candidates, clonotype clustering, junctional sequence parsing, parallel lineage mining, public clonotype search results — these names are treated as supportive analysis objects by default; do not include them all just because CDRH3 or partial sequences appear
   - Only include when these clones are explicitly tested/characterized as antibodies in the body text or dedicated quantitative tables (with independent kinetics / binding / functional / structure conclusions)
8. **Control rows / inferred germline / previously published reference panels**: If tables or figure legends explicitly state `Control mAbs`, `control Abs`, `previously published antibodies`, `inferred germline`, `iGL`, `other representative antibodies published previously`, etc., these records do not enter the antibodies array by default

**For non-binding antibodies with sequence data** (e.g., although binding experiments are negative or the antibody was not selected for in-depth characterization, CDRH3/VH/VL sequence information was successfully obtained), they should still be included. Note the target attribution in Target_Name (use the table value if the table has a target column; otherwise use the common target of the same panel in the paper).

## Key Operational Points

1. Scan the full text, identify the paper's newly characterized core antibodies (excluding failed candidates, control antibodies, commercially known antibodies)
   - First ask yourself: does this name represent "an entity that independently exists and is described by the paper as a native/full antibody"? If it is only `L9H-F10K`, `F10H-L9K`, `MRCA`, `D2H`, `F10K` or similar chain source/recombination labels, exclude by default
   - If the main narrative compares "native antibody" vs "engineered swap / inferred ancestor," the final antibodies array should only retain native antibodies; other constructs serve only as internal evidence for explanation
   - If the main narrative/figure legends focus on `sequence alignment`, `public clonotype search`, `control mAbs`, `parallel lineage`, `junctional sequence analysis`, `inferred germline/iGL`, first determine which names are genuinely tested or produced antibodies of this paper; comparator/control/candidate clones not independently supported by body text or dedicated quantitative tables should not be included
2. Antibody_Type and Antibody_Isotype must be strictly distinguished:
   - `Antibody_Type` should contain the antibody format/category, such as `mAb`, `Fab`, `scFv`, `VHH`, `bispecific antibody`
   - `Antibody_Isotype` should contain the immunoglobulin subtype, such as `human IgG1`, `mouse IgG2a`, `IgM`
   - Do not mistakenly fill `IgG` / `IgG1` into `Antibody_Type`; if the text explicitly says it is a monoclonal antibody and the subtype is separately stated, prioritize writing `mAb`
3. Target_Name must include specific variant information: e.g., "SARS-CoV-2 Spike RBD (BA.4/5)"
   - When information necessary to distinguish records involves strain, variant, isolate, cell line, host background, antigen preparation form, construct, or batch, and the schema has no independent field for it, incorporate this conditional information into `Target_Name` in parentheses.
   - Examples: `Plasmodium falciparum parasite (3D7)`, `SARS-CoV-2 Spike RBD (BA.4/5)`, `MPXV A35 (clade IIb)`, `HIV-1 Env trimer (BG505 SOSIP)`.
   - However, if the main subject is "this antibody targets SARS-CoV-2" and other viruses are only cross-reactivity subjects, do not expand Cross_Reactivity into multiple Target_Name records
   - `Target_Type` should preferentially state the biological class, such as `Viral Protein`, `Bacterial Protein`, `Tumor antigen`, `Cell-surface receptor`; do not write only `surface protein`, `membrane protein` which lack source category information
4. **vh_sequence_aa / vl_sequence_aa fields are extremely important**:
  - These two fields only allow **complete variable region amino acid sequences**; germline gene names, V/J annotations, identity percentages, or any abbreviations are not allowed.
  - If the text or images only show `IGHV1-2`, `IGKV3-20`, `VH4-4/JH3`, `V identity 93%` or similar germline / gene typing information without a complete amino acid string, then `vh_sequence_aa` / `vl_sequence_aa` must be left empty.
  - Do not disguise germline information as complete sequences; this will directly cause evaluation errors.
  - **Do not** create separate VH_Germline or VL_Germline fields.
  - If the original text directly provides the complete sequence, you must write the real sequence into the corresponding flat field; do not leave it empty.
  - If the original text uses three-letter amino acid notation, such as `Glu Val Gln Leu...`, `Ile-Leu-Pro-Arg...`, you must first convert to standard single-letter sequence before writing into the corresponding field; do not output raw three-letter tokens.
  - If supplementary tables / OCR tables / markdown have already transcribed complete sequences, even if from supplementary material, treat them as "directly provided complete sequence"; do not downgrade to `API Fetch`.
